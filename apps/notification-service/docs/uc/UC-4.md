# UC-4. 알림채널 설정 — 리뷰·연구 노트

사용자가 개인 알림 수신 채널(SMS/Email/알림톡)을 켜고 끄는 흐름입니다. UC 명세는 [../02-actors-usecases.md](../02-actors-usecases.md) §UC-4에 있습니다.

- **상태**: 부분 존재 — 데이터(`ChannelSetting` 복합키 Entity)와 조회(`ChannelSettingService`)는 UC-1이 이미 씀. REST CRUD 계약(GET/PUT)은 미구현
- **관련 FR**: FR-12
- **주 액터**: 사용자 (FE)
- **미니 방침**: 프론트 최소화 — REST 계약(요청/응답)만, 화면은 만들지 않음

---

## 1. 설계 초점

| 기술 | UC-4에서의 역할 |
|------|----------------|
| JPA 복합키 | (userId, channelType) 단위 설정 저장 |
| 인증 예외 | 토큰 없이 접근해야 하는 설정 조회 경로 |

---

## 2. 코드 리뷰 렌즈

**UC-1과의 연결**이 핵심입니다: UC-4가 저장하는 `ChannelSetting`을 UC-1의 발송 파이프라인(FR-3)이 **읽어서** enabled=false인 채널을 거릅니다. 즉 UC-4는 UC-1의 입력 데이터를 만드는 쪽.

- 볼 것 (구현 시): GET(조회)/PUT(저장) 계약, 저장 시 **캐시 무효화**(`@CacheEvict`)를 UC-1의 Caffeine 캐시에 걸어야 옛 설정으로 발송하지 않는다 — 이게 UC-1 리뷰 렌즈의 "캐시 무효화 없음" 지적과 직결된다.

---

## 3. 직접 관찰·실측법 (구현 후 채움)

- PUT으로 설정을 끈 직후 UC-1 발송을 돌려, 캐시가 무효화돼 새 설정이 반영되는지(또는 TTL 만료까지 옛 값인지) 확인.
