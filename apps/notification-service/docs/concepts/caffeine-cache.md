# Caffeine 캐시와 Spring 캐시 추상화

> 캐시는 "같은 질문에 두 번 DB를 치지 않겠다"는 결정이고, 그 대가로 "옛 값을 볼 수 있는 시간"을 삽니다. 이 문서는 UC-1(조회)과 UC-4(갱신)가 공유하는 캐시의 동작 원리와, 그 경계가 어디서 끝나는지를 다룹니다.

근거: UC-1 실측(2026-07-20 — 같은 키 2번째 조회에 미스 로그 없음), UC-4 스모크(2026-07-21 — PUT 직후 재발행 차단). 실행으로 보지 않은 것은 각 절에서 밝힙니다.



## 1. 로컬 인메모리 캐시 — 무엇을 사고 무엇을 포기하는가

> Caffeine은 앱 프로세스의 힙 안에 사는 Map에 가깝습니다. 네트워크 없이 나노초 단위로 읽는 속도를 사는 대신, "이 프로세스만 아는 값"이라는 한계를 함께 삽니다.

Caffeine은 자바 진영의 표준 로컬 캐시 라이브러리입니다. Guava Cache의 후속으로, 접근 빈도와 최근성을 함께 보는 W-TinyLFU 알고리즘으로 퇴거(eviction) 대상을 고릅니다. Redis 같은 외부 캐시와의 근본 차이는 위치입니다 — 값이 앱 힙 안에 있으므로 조회에 네트워크 왕복이 없고, 대신 프로세스가 죽으면 함께 사라지며 다른 인스턴스와 공유되지 않습니다(§5).

이 프로젝트에서 캐시하는 것은 채널 수신 설정 `(userId, channelType) → boolean`입니다. 발송 파이프라인이 수신자마다 이 조회를 하므로 읽기는 많고, 본인이 설정 화면에서 바꿀 때만 쓰기가 일어나므로 쓰기는 드뭅니다. 읽기 편중 + 작은 값 + 약간의 신선도 양보 허용 — 로컬 캐시에 이상적인 조건입니다.



## 2. 캐시 스펙 — 시간과 크기의 상한

> 캐시는 반드시 버리는 규칙과 함께 삽니다. 상한 없는 캐시는 메모리 누수의 다른 이름입니다.

스펙은 `send/config/CacheConfig.java`에서 정의합니다.

```java
cacheManager.setCaffeine(Caffeine.newBuilder()
        .expireAfterWrite(Duration.ofMinutes(10))   // 쓴 지 10분이 지나면 퇴거 (TTL)
        .maximumSize(10_000)                        // 항목 1만 개 상한 — 넘으면 W-TinyLFU가 선별 퇴거
        .recordStats());                            // 히트율 통계 수집
```

`expireAfterWrite`는 "마지막으로 **쓴** 시점" 기준입니다. 조회할 때마다 수명이 연장되는 `expireAfterAccess`와 다릅니다 — 설정처럼 원본이 바뀔 수 있는 데이터는 접근이 잦다고 수명을 늘려주면 옛 값이 영원히 살아남을 수 있어, 쓰기 기준 만료가 안전합니다.



## 3. 어노테이션 3형제 — @Cacheable, @CachePut, @CacheEvict

> 셋 다 "메서드 주변에서 캐시를 조작하는 프록시 지시문"이지만, 메서드 실행 여부와 캐시에 하는 일이 다릅니다.

| 어노테이션 | 메서드 실행 | 캐시에 하는 일 | 이 프로젝트의 사용처 |
|---|---|---|---|
| `@Cacheable` | 캐시 미스일 때만 | 미스면 리턴값 저장, 히트면 메서드 건너뛰고 캐시값 반환 | `isEnabled` 조회 (UC-1) |
| `@CachePut` | **항상** | 리턴값을 해당 키에 덮어씀 | `update` 저장 (UC-4) |
| `@CacheEvict` | 항상 | 해당 키(또는 전체)를 삭제 | 미사용 — 다음 조회에 미스 비용이 발송 경로에 청구되어 put을 선택 |

세 어노테이션 모두 Spring AOP 프록시가 처리합니다. 따라서 두 가지 성질이 따라옵니다.

첫째, **캐시 기록은 메서드(트랜잭션 포함)가 정상 리턴한 뒤** 프록시가 수행합니다. 예외가 나면 캐시는 건드리지 않습니다 — Kafka 리스너의 "정상 리턴 = 커밋"([kafka-message-handling §2](kafka-message-handling.md#2-언제-완료로-치는가--auto-commit과-ack-mode))과 같은 결의 구조입니다.

둘째, 같은 클래스 안에서 `this.isEnabled(...)`로 부르면 프록시를 우회해 캐시가 조용히 무시됩니다. 이 self-invocation 함정은 CircuitBreaker에서 실측으로 확인한 것과 동일한 원리입니다([external-call-and-resilience §4](external-call-and-resilience.md#4-어노테이션이-조용히-무력해지는-경우--self-invocation)).



## 4. 키는 계약이다 — 조회와 갱신의 SpEL은 글자까지 같아야 한다

> 캐시 키는 컴파일러가 검사하지 않는 문자열 계약입니다. 어긋나면 에러 없이, 갱신이 "다른 서랍에 넣기"가 됩니다.

조회와 저장은 같은 키 SpEL을 씁니다(`channel/application/ChannelSettingService.java`).

```java
@Cacheable(cacheNames = "channelSetting", key = "#userId + ':' + #channelType")  // 조회
@CachePut(cacheNames = "channelSetting", key = "#userId + ':' + #channelType")   // 저장
```

만약 한쪽 키가 달라지면 — 예를 들어 구분자 오타 — 예외도 기동 실패도 없습니다. 저장된 새 값은 아무도 읽지 않는 고아 항목이 되고, 조회는 옛 키의 값이 TTL로 죽을 때까지 그것을 계속 반환합니다. `@CachePut`이 막으려던 "수신 거부했는데 10분간 알림이 온다"가 로그 한 줄 없이 재발하는 셈입니다. 이 시나리오는 코드상 추론이며 실행으로 재현하지는 않았습니다([UC-4 후속 검증](../learning/UC-4-channel-setting.md#후속-검증)).



## 5. 로컬 캐시의 경계 — 인스턴스가 2대가 되는 순간

> @CachePut은 요청을 받은 프로세스의 힙만 갱신합니다. 다른 인스턴스는 갱신 사실 자체를 모릅니다.

앱을 2대로 늘리면, PUT이 A 인스턴스에 도착해 A의 캐시는 즉시 새 값이 되지만 **B는 자기 캐시의 옛 값을 TTL까지 씁니다.** B 입장에선 아무 일도 없었으니까요. 해법은 셋 중에서 고릅니다(다중 기동 실험은 하지 않았으므로 구조적 추론입니다).

| 방향 | 효과 | 비용 |
|---|---|---|
| 공유 캐시 (Redis 등) | 모든 인스턴스가 같은 값 | 조회마다 네트워크 홉 — 로컬 히트 속도 상실 |
| 이벤트 기반 무효화 | 로컬 속도 유지, 변경 이벤트로 각자 evict | 발행·구독 인프라와 복잡도 |
| 짧은 TTL 감수 | 코드 무변경 | 불일치 창이 TTL만큼 상존 |

실무에서는 로컬(1차) + Redis(2차)를 겹치는 2단 캐시로 속도와 일관성을 절충하기도 합니다. 현재는 단일 인스턴스 전제라 TTL 10분 + `@CachePut`으로 충분합니다.
