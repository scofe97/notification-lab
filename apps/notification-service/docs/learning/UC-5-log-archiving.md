# UC-5 · 로그 아카이빙 — @Scheduled 배치, 전일 이력 export

> 스케줄러가 전일 발송 이력을 OpenSearch에서 읽어 파일로 내보내는 흐름(FR-8·FR-11)을 학습 단위로 다룹니다. 사람 요청이 아니라 시간이 트리거라는 점, 그리고 배치 실패·중복 실행을 어떻게 다루는지가 핵심입니다.

- 상태: **미구현** — `notification.history` 패키지 미생성. 이 문서는 구현을 끝낸 뒤 채웁니다.
- 근거(예정): 구현 커밋, 스모크 실측, [UC-5 리뷰 노트](../uc/UC-5.md)
- 연결: [UC-1 학습 문서](UC-1-kafka-notification.md)(이력을 만드는 발송 흐름) · [UC-3 학습 문서](UC-3-history-query.md)(이 색인을 조회하는 쪽) · [컨벤션](../../../../AGENTS.md)
- 학습 순서: 선행 → UC-1(발송 이력 생산). 후속 → UC-3(색인 조회). 진행 순서 SSOT는 [ROADMAP](../../../../ROADMAP.md).



## 구현 개요 — 먼저 읽는 지도 (구현 후 채움)

> 구현을 끝내면 아래 표에 실제 파일과 역할을 채웁니다. 지금은 설계 초점만 예측으로 둡니다.

구현 예정 범위는 셋입니다. ① `@Scheduled` cron이 정해진 시각에 배치를 시작, ② 전일 인덱스 선택과 export 파일 형식, ③ 배치 실패·중복 실행(멱등성) 처리. 전부 헥사고날 `history` 컨텍스트에 담을 예정입니다([AGENTS.md](../../../../AGENTS.md) 컨벤션).

| 구성요소 | 파일 (예정) | 역할 |
|---|---|---|
| 스케줄 트리거 | `history/application/...Scheduler` | `@Scheduled` cron으로 배치 진입 |
| 전일 인덱스 선택 | `history/application/...` | 어제 날짜 인덱스를 골라 조회 |
| out-port | `history/domain/port/out/...` | 이력 조회·export 인터페이스 — 도메인이 선언 |
| export 어댑터 | `history/infrastructure/...` | out-port 구현, OpenSearch 조회 후 파일 쓰기 |

## 후속 검증 (구현 후 채움)

| 항목 | 상태 | 확인 방법 |
|---|---|---|
| cron 트리거 관찰 | 대기 | cron을 짧은 주기로 임시 변경해 트리거가 도는지, export 파일이 생기는지 확인 (후 원복) |
| 중복 실행 방지 | 대기 | 같은 날짜에 두 번 돌 때 export가 중복되지 않는지(멱등) 확인 |
| UC-3 조회 대상과 일치 | 대기 | export 파일 내용이 UC-3 조회 대상 색인과 일치하는지 대조 |



## Phase 진행 기록

> 각 Phase는 구현 후 대화로 진행하고, 여기에는 통과 여부와 해결된 오해만 남깁니다. 개인 답변 원문은 기록하지 않습니다.

- [ ] Phase 1 · 맥락과 예측 — 시간이 액터라는 점과 배치 실패·중복 실행의 위험을 예측한다
- [ ] Phase 2 · 안내된 흐름 읽기 — cron 트리거 → 전일 인덱스 선택 → export까지 설명한다
- [ ] Phase 3 · 실패·경계 추적 — 배치 도중 실패·재실행·빈 인덱스의 처리를 증거 등급과 함께 설명한다
- [ ] Phase 4 · 실측 실습 (선택) — cron 단축으로 트리거와 export 파일 생성을 관찰한다
- [ ] Phase 5 · 능동 인출 — 5문항과 자기 설명을 통과한다
