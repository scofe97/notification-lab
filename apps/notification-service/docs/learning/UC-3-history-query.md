# UC-3 · 알림 이력 조회 — OpenSearch 색인 조회, 채널별 쿼리 매퍼

> 사용자가 채널·기간으로 발송 이력을 검색하는 흐름(FR-10)을 학습 단위로 다룹니다. UC-5 아카이빙이 색인한 이력을 읽는 쪽이라, 색인 구조를 먼저 이해한 뒤 조회 쿼리 조립을 봅니다.

- 상태: **미구현** — `notification.history` 패키지 미생성. 이 문서는 구현을 끝낸 뒤 채웁니다.
- 근거(예정): 구현 커밋, 스모크 실측, [UC-3 리뷰 노트](../uc/UC-3.md)
- 연결: [UC-5 학습 문서](UC-5-log-archiving.md)(이력을 색인하는 쪽) · [UC-1 학습 문서](UC-1-kafka-notification.md)(이력을 만드는 발송 흐름) · [컨벤션](../../../../AGENTS.md)
- 학습 순서: 선행 → UC-5(색인 구조). 진행 순서 SSOT는 [ROADMAP](../../../../ROADMAP.md).



## 구현 개요 — 먼저 읽는 지도 (구현 후 채움)

> 구현을 끝내면 아래 표에 실제 파일과 역할을 채웁니다. 지금은 설계 초점만 예측으로 둡니다.

구현 예정 범위는 셋입니다. ① `GET .../history?channel=&from=&to=` REST 계약, ② 채널마다 다른 쿼리 형식을 고르는 매퍼(전략/팩토리), ③ OpenSearch 조회와 이력 응답 형상. 전부 헥사고날 `history` 컨텍스트에 담을 예정입니다([AGENTS.md](../../../../AGENTS.md) 컨벤션).

| 구성요소 | 파일 (예정) | 역할 |
|---|---|---|
| REST 어댑터 | `history/api/...` | 채널·기간 조건으로 조회 요청 접수 |
| 쿼리 매퍼 | `history/application/QueryParamMapperFactory` | 채널로 매퍼를 골라 채널별 쿼리 형식 캡슐화 |
| out-port | `history/domain/port/out/...` | 이력 조회 인터페이스 — 도메인이 선언 |
| OpenSearch 어댑터 | `history/infrastructure/...` | out-port 구현, OpenSearch 쿼리 조립·색인 조회 |

## Phase 진행 기록

> 각 Phase는 구현 후 대화로 진행하고, 여기에는 통과 여부와 해결된 오해만 남깁니다. 개인 답변 원문은 기록하지 않습니다.

- [ ] Phase 1 · 맥락과 예측 — 채널별 쿼리 분기가 왜 매퍼(전략)로 캡슐화되는지 예측한다
- [ ] Phase 2 · 안내된 흐름 읽기 — 요청 → 매퍼 선택 → OpenSearch 쿼리 → 응답까지 설명한다
- [ ] Phase 3 · 실패·경계 추적 — 빈 결과·잘못된 기간·색인 부재의 전파를 증거 등급과 함께 설명한다
- [ ] Phase 4 · 실측 실습 (선택) — 채널별 쿼리 차이를 OpenSearch 로그로 확인한다
- [ ] Phase 5 · 능동 인출 — 5문항과 자기 설명을 통과한다
